### 🚨 `完了` Status Option Missing from Project Edit Form
- **Severity:** High
  - High: Broken functionality — users have no UI path to mark a project as 完了. The `完了` status exists in the DB and is accepted by the API, but the select element omits it entirely.
- **Steps to Reproduce:**
  1. Navigate to 案件管理 page.
  2. Click the 編集 button on any project row.
  3. Open the ステータス dropdown (`#prj-status`).
  4. Observe available options.
- **Observed Behavior:** The `#prj-status` select renders exactly 6 options: 初回訪問済, 現地調査済, 商談中, 見積提出済, 契約済, 失注. The `完了` value is absent. Playwright's `page.selectOption('#prj-status', '完了')` timed out after 30 seconds with "did not find some options". DB query confirmed the newly created 田中様邸 project remained at `status='契約済'` after the 完了 edit attempt, because the form was submitted with the pre-filled value. Additionally, the project filter bar (`#prj-filter`) has no `data-val="完了"` button, so existing 完了 projects (imported from Excel) cannot be filtered.
- **Expected Behavior:** 完了 should be a selectable status in the edit form, representing a project that has been delivered and closed. This is a standard business workflow status (初回訪問済 → 商談中 → 契約済 → 完了). The `GET /api/projects/by-status` response already includes a 完了 bucket. The server's pipeline query correctly excludes it: `WHERE status NOT IN ('契約済','失注','完了')`.
- **Developer Guide:** In `dashboard/public/index.html`, line 720–727, add `<option value="完了">完了</option>` to the `#prj-status` select. Also add a filter button at line 466–473: `<button class="filter-btn" data-val="完了">完了</button>`. Consider placing 完了 after 契約済 to reflect workflow order. Note: the server pipeline SQL already handles 完了 correctly — only the frontend form needs updating.

### 🚨 Monthly Chart Label Malformed for ISO-Format Dates Entered via CRUD
- **Severity:** High
  - High: Chart data corruption — when any project is created or edited via the CRUD form and saved with a `contract_date` in ISO format (`YYYY-MM-DD`), the monthly chart aggregation query produces a malformed label (e.g., `"-2026"`) and groups all such projects under a single broken bucket, making the chart unreadable.
- **Steps to Reproduce:**
  1. Navigate to 案件管理, open ＋ 新規案件を追加.
  2. Fill in 案件名, 顧客名, ステータス=契約済, 契約日=2026-06-25 (from HTML date picker), 契約金額=1950000. Save.
  3. Navigate to ダッシュボード.
  4. Observe the 月別 受注金額 chart, or call `GET /api/projects/monthly` directly.
- **Observed Behavior:** `GET /api/projects/monthly` returned `{"month":"-2026","count":1,"total":1950000}` for the newly created project. This label is malformed and does not sort correctly with valid months like `"2024-03"`. The root cause: the SQL in `server.js` lines 86–92 uses `SUBSTR(contract_date, 1, instr(contract_date, '/') - 1)` to extract the year from the `YYYY/M/D` format used by imported data. For ISO dates like `"2026-06-25"`, `instr('2026-06-25', '/')` returns 0, so `SUBSTR('2026-06-25', 1, -1)` returns an empty string `""`. The month token extraction then evaluates `CAST('2026-06-25' AS INTEGER)` = `2026`, producing `printf('%s-%02d', '', 2026)` = `"-2026"`. Live API log excerpt: `[{"month":"-2026","count":1,"total":1950000},{"month":"2024-03","count":2,"total":2170000},...]`
- **Expected Behavior:** All projects with `contract_date` in either `YYYY/M/D` or `YYYY-MM-DD` format should produce a correctly formatted `YYYY-MM` label. The new project (契約日 2026-06-25) should appear as `"2026-06"` with its 1,950,000 yen contribution.
- **Developer Guide:** In `dashboard/server.js` lines 83–93, replace the current `printf` approach with a format-agnostic one. SQLite's `strftime('%Y-%m', ...)` cannot parse `YYYY/M/D` directly, but a `REPLACE` pre-processing step can normalize both formats: `strftime('%Y-%m', REPLACE(REPLACE(contract_date, '/', '-'), ...))`. Alternatively, store dates in ISO format consistently (address in `db.js` import). The simplest fix that handles both: use `CASE WHEN contract_date LIKE '%/%' THEN printf('%s-%02d', substr(contract_date, 1, 4), cast(substr(contract_date, 6, 2) as integer)) ELSE substr(contract_date, 1, 7) END as month`.

### 🚨 Body Scroll Not Locked When Modal Is Open — Submit Button Click Intercepted
- **Severity:** Medium
  - Medium: UI interaction broken in automated testing context; poses real usability risk on long pages where background content can scroll beneath the modal overlay.
- **Steps to Reproduce:**
  1. Navigate to 顧客台帳 (which already has 35+ rows making the page scrollable).
  2. Open the ＋ 新規顧客を追加 modal.
  3. While the modal is open, check `document.body.style.overflow` or `window.getComputedStyle(document.body).overflow`.
  4. Attempt to scroll the main content area using keyboard or trackpad while the modal is open.
  5. In an automated Playwright test: call native `page.click()` on the submit button (not `page.evaluate(() => btn.click())`).
- **Observed Behavior:** `window.getComputedStyle(document.body).overflow` returns `"visible"` while the modal is open. The `<main class="main">` element remains scrollable. In Playwright's native click, `scrollIntoView` on the submit button at the bottom of the form causes the underlying page to scroll, repositioning a `<td>03-3960-5566</td>` from the customer table to the same visual coordinates as the button. Playwright logs: `"<td>03-3960-5566</td> from <main class="main">…</main> subtree intercepts pointer events"`. The click fails with a 30-second timeout. Native browser user clicking with a mouse is unaffected in most cases (mouse targeting is precise), but keyboard/scroll interaction creates visual confusion.
- **Expected Behavior:** When any `.modal-overlay.open` is active, `document.body.overflow` should be set to `hidden` (or `clip`) to prevent background content from scrolling. This is standard modal UX behavior.
- **Developer Guide:** In `dashboard/public/index.html`, modify `openCustomerModal()`, `openProjectModal()`, `openInvoiceModal()`, `openEmployeeModal()`, and `openStaffModal()` (lines ~1476, ~1558, ~1749, ~1655, ~1294) to add `document.body.style.overflow = 'hidden'` when opening. In `closeFormModal(id)` at line 1464, add `document.body.style.overflow = ''` to restore. Alternatively, add a CSS rule: `.modal-overlay.open ~ * { overflow: hidden }` — though JS is more reliable across all modal types. A cleaner approach: add/remove a CSS class `body.modal-open { overflow: hidden }` toggled by the modal open/close functions.

### 🚨 All Three "Add" Buttons Lack `id` Attributes
- **Severity:** Medium
  - Medium: Reduces testability and violates WCAG accessibility guidelines for interactive controls. Not a runtime crash but blocks reliable E2E test automation and assistive technology targeting.
- **Steps to Reproduce:**
  1. Navigate to 顧客台帳, inspect the ＋ 新規顧客を追加 button.
  2. Check `document.querySelector('#page-customers .add-btn').id`.
  3. Repeat for 案件管理 and 請求・入金管理 pages.
- **Observed Behavior:** All three add buttons (`#page-customers .add-btn`, `#page-projects .add-btn`, `#page-invoices .add-btn`) return `id = ""` (empty string). The buttons are identified only by class and inline `onclick` handlers: `<button class="add-btn" onclick="openCustomerModal()">＋ 新規顧客を追加</button>`. Any automated test or QA tool attempting `page.click('#btn-add-customer')` will fail with "element not found". Playwright selector logs: `page.waitForSelector: Timeout 10000ms exceeded — waiting for locator('#btn-add-customer') to be visible`.
- **Expected Behavior:** Each add button should have a unique, descriptive `id` attribute (e.g., `id="btn-add-customer"`, `id="btn-add-project"`, `id="btn-add-invoice"`) enabling stable CSS/JS/test targeting.
- **Developer Guide:** In `dashboard/public/index.html`, update: line 464 add `id="btn-add-project"`, line 518 add `id="btn-add-invoice"`, line 542 add `id="btn-add-customer"`. Also consider replacing the `onclick` inline handlers with `addEventListener` in the script section for separation of concerns.

### 🚨 `/api/projects/by-status` Returns `estimate_amount` Totals, Not `contract_amount`
- **Severity:** Medium
  - Medium: Misleading dashboard chart data. For 契約済 projects, the chart shows estimated amounts instead of contracted amounts, causing a visible discrepancy between the KPI card total and the chart bar total.
- **Steps to Reproduce:**
  1. Create a project with 見積金額 (estimate) = 1,980,000 and 契約金額 (contract) = 1,950,000 and status = 契約済.
  2. Call `GET /api/summary` and note `contracted_total`.
  3. Call `GET /api/projects/by-status` and note `total` for the 契約済 bucket.
  4. Compare the two values.
- **Observed Behavior:** After creating 田中様邸 (estimate=1,980,000, contract=1,950,000, status=契約済): `GET /api/summary` returned `contracted_total=12,260,000`. `GET /api/projects/by-status` returned `{"status":"契約済","count":15,"total":12,290,000}`. The 30,000 yen discrepancy is exactly `1,980,000 - 1,950,000` — the 田中様邸 estimate-vs-contract difference. The SQL in `server.js` line 67 reads: `SELECT status, COUNT(*), COALESCE(SUM(estimate_amount),0) as total FROM projects GROUP BY status`. It uses `estimate_amount` across all statuses.
- **Expected Behavior:** For 契約済 projects, the by-status API should use `contract_amount` (since a contract has been signed). Pre-contract statuses may reasonably use `estimate_amount`. The chart title "案件ステータス別" should reflect contracted value for 契約済 rows to match the KPI card.
- **Developer Guide:** In `dashboard/server.js` line 67, change `SUM(estimate_amount)` to `SUM(COALESCE(contract_amount, estimate_amount))` — this uses `contract_amount` when available and falls back to `estimate_amount` for pre-contract stages. Or use a `CASE WHEN status='契約済' THEN contract_amount ELSE estimate_amount END` expression for precise per-status logic.

### 🚨 `inv-billing_type` Field Label and Placeholder Conflict With Actual Data Semantics
- **Severity:** Low
  - Low: UX confusion and potential data inconsistency. The field label "区分" with placeholder "例：工事費" implies a billing category (expense type), while the DB column `billing_type` stores payment structure values like "一括" (lump sum) imported from Excel. New entries created by users following the placeholder will create inconsistent data.
- **Steps to Reproduce:**
  1. Navigate to 請求・入金管理 and open the ＋ 新規請求を追加 modal.
  2. Observe the second field labeled "区分" with placeholder "例：工事費".
  3. Compare with existing invoice rows in the table, which show "一括" in the 区分 column.
- **Observed Behavior:** The field label reads "区分" and the placeholder reads "例：工事費" (suggesting billing category). However, existing DB records imported from Excel store `billing_type` as "一括" (lump-sum payment type), "前払い", or similar payment structure terms. Playwright test confirmed: the test created an invoice with `billing_type=null` (field was left empty by the test code because no structured input was available), while existing real invoices have `billing_type="一括"`. Live DB query: `billing_type=null` for INV-2026-0036 vs `billing_type="一括"` for INV-2026-0035.
- **Expected Behavior:** The label should read "請求種別" or "支払方法" with a placeholder "例：一括" or be implemented as a `<select>` with options: 一括, 分割 (前払い/後払い). If the column truly means billing category, the existing imported data should be corrected.
- **Developer Guide:** In `dashboard/public/index.html` line 838, change `<label class="form-label">区分</label>` to `<label class="form-label">請求種別</label>`. Change line 839's input to a `<select>` with options matching actual DB values: `<select class="form-select" id="inv-billing_type"><option value="">選択してください</option><option value="一括">一括</option><option value="分割">分割</option></select>`. Update `openInvoiceModal()` at line 1767 to use `document.getElementById('inv-billing_type').value = inv.billing_type || ''`.

### 🚨 No `完了` Filter Button in Project Filter Bar
- **Severity:** Low
  - Low: UX gap. Projects with `完了` status (which exist in the imported dataset) are only visible when the "すべて" filter is active. There is no dedicated filter to isolate completed projects.
- **Steps to Reproduce:**
  1. Navigate to 案件管理.
  2. Observe the filter bar buttons: すべて, 契約済, 見積提出済, 商談中, 現地調査済, 初回訪問済, 失注.
  3. Note the absence of a 完了 filter button.
  4. Call `GET /api/projects/by-status` — confirm `{"status":"完了","count":1}` is returned (verified during test).
- **Observed Behavior:** The filter bar in `#prj-filter` contains 7 buttons (all + 6 statuses) but omits `完了`. Users cannot filter to see only completed projects. The by-status chart does include a `完了` bar, creating an inconsistency between chart and filter capabilities.
- **Expected Behavior:** A `完了` filter button should exist in the filter bar, consistent with how all other active statuses have filter buttons. This is especially important once Issue #1 (missing 完了 select option) is fixed.
- **Developer Guide:** In `dashboard/public/index.html`, line 473 (after the 失注 filter-btn), add: `<button class="filter-btn" data-val="完了">完了</button>`. This requires no backend changes — `renderProjects()` already uses `data-val` for filtering via `allProjects.filter(p => !status || p.status === status)`.

## Recommendations

Top 3 highest-priority fixes in priority order:

1. **[High — Bug #1] Add `完了` to the project status select** (`dashboard/public/index.html` line 726): Add `<option value="完了">完了</option>` to `#prj-status`. Without this, the company president has no UI path to close out completed projects. The entire business lifecycle scenario breaks at Step 5. Also add the matching filter button (Bug #7) in the same commit.

2. **[High — Bug #2] Fix monthly chart SQL for ISO-date `contract_date` values** (`dashboard/server.js` lines 83–93): The current `SUBSTR/instr` approach only works for `YYYY/M/D` format. Any project saved via the CRUD form (which stores HTML date picker values as `YYYY-MM-DD`) produces a malformed `"-2026"` label that corrupts the chart. Replace with a format-agnostic expression using `CASE WHEN contract_date LIKE '%/%'` branching, or normalize dates to ISO format at insert time in the POST/PUT handlers.

3. **[Medium — Bug #3] Lock body scroll when modal is open** (`dashboard/public/index.html`): Add `document.body.style.overflow = 'hidden'` in all modal-open functions and restore `overflow = ''` in `closeFormModal()`. This prevents background content from scrolling under the modal overlay, fixes the Playwright native click intercept (which would affect any automation testing of this dashboard), and aligns with standard accessible modal UX patterns.
