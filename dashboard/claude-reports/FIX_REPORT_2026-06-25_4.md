## 修正詳細

### Fix #1 — `完了` ステータスの案件がパイプライン KPI に集計される
- **File:** `dashboard/server.js`
- **Lines affected:** 36–38（`/api/summary` ルート内 pipeline クエリ）
- **Root cause:** pipeline 集計クエリの除外条件が `status NOT IN ('契約済','失注')` となっており、受注後に完了した案件（`完了`）が除外されていなかった。そのため `完了` 案件の `estimate_amount` と件数が、進行中（商談・交渉中）案件を表すはずの `pipeline_count` / `pipeline_total` に二重・誤集計され、経営判断用の KPI カードが実態より大きい値を示していた。
- **Fix applied:** NOT IN リストに `'完了'` を追加。

  Before:
  ```sql
  SELECT COUNT(*) as count, COALESCE(SUM(estimate_amount),0) as total
  FROM projects WHERE status NOT IN ('契約済','失注')
  ```
  After:
  ```sql
  SELECT COUNT(*) as count, COALESCE(SUM(estimate_amount),0) as total
  FROM projects WHERE status NOT IN ('契約済','失注','完了')
  ```
- **Side-effect check:** 変更は pipeline クエリ 1 行のみ。`contracted`（`status='契約済'`）、`lost`（`status='失注'`）、`totalProjects`、`invoiceSummary` の各集計は独立しており影響なし。`完了` は元々 pipeline に含まれるべきでないステータスのため、進行中案件の集計には回帰なし。

### Fix #2 — `probability` フィールドが `"100.0"` 文字列で返される
- **File:** `dashboard/server.js`
- **Lines affected:** 98–110（`GET /api/projects` ルート）
- **Root cause:** SQLite が `probability` を REAL（浮動小数点）型で格納するため、`better-sqlite3` が整数値 `100` を `100.0` として返していた。フロントエンドの案件テーブルに `100.0` と表示され、編集モーダルの確度入力欄にも `100.0` がセットされて再保存ループの懸念があった。
- **Fix applied:** SELECT で `CAST(probability AS TEXT) as probability` により文字列化したうえで、レスポンスの `.map()` 内で `.replace(/\.0$/, '')` を適用し、末尾の `.0` をトリミング。`null` の場合はそのまま保持。

  Before:
  ```js
  let sql = "SELECT * FROM projects WHERE 1=1";
  ...
  const rows = db.prepare(sql).all(...params);
  res.json(rows);
  ```
  After:
  ```js
  let sql = "SELECT *, CAST(probability AS TEXT) as probability FROM projects WHERE 1=1";
  ...
  const rows = db.prepare(sql).all(...params).map(r => ({
    ...r,
    probability: r.probability ? r.probability.replace(/\.0$/, '') : r.probability,
  }));
  res.json(rows);
  ```
- **Side-effect check:** `null` の確度は `r.probability ? ... : r.probability` の三項演算で安全に保持される。`SELECT *` の後に `CAST(...) as probability` を置くことで元の `probability` カラムを上書きしているため、重複キーは発生しない。`.replace(/\.0$/, '')` は末尾 `.0` のみを対象とするため `12.5` 等の小数値は変化しない（本フィールドは整数運用のため実害なし）。**Falsy 注意点:** `probability = 0`（確度 0%）の場合、CAST 後の文字列は `"0"`（truthy）のため `.replace` が正しく適用され、`0` が欠落することはない。

### Fix #3 — 文字化けテキスト（エンコーディング不一致）がそのまま保存される
- **File:** `dashboard/server.js`
- **Lines affected:** 11–23（`express.json()` の直後に追加された検証ミドルウェア）
- **Root cause:** `Content-Type: application/json`（charset 指定なし）で UTF-16 等の非 UTF-8 日本語テキストを送信すると、デコード失敗により `?` の連続（`?? ??`・`?????2-14-5` 等）に化けたまま DB に保存され、サーバーは `{"ok":true}` を返していた。データ整合性が破壊されていた。
- **Fix applied:** `express.json()` の直後・全ルート定義の前に検証ミドルウェアを追加。対象テキストフィールドに 2 文字以上連続する `?`（`/\?{2,}/`）が含まれる場合、HTTP 400 と日本語エラーメッセージを返してリクエストを早期に拒否する。

  ```js
  const GARBLE_RE = /\?{2,}/;
  const TEXT_FIELDS = ['name','address','note','staff','customer_name','project_name','building_type','source','work_type','billing_type'];
  app.use((req, res, next) => {
    if (req.body && typeof req.body === 'object') {
      for (const key of TEXT_FIELDS) {
        if (typeof req.body[key] === 'string' && GARBLE_RE.test(req.body[key])) {
          return res.status(400).json({ error: `フィールド「${key}」の文字コードが不正です。UTF-8で送信してください。` });
        }
      }
    }
    next();
  });
  ```
- **Side-effect check:** ミドルウェアは `express.json()` の後・`express.static()` および全 API ルートの前に配置されており、すべての POST/PUT リクエストを通過時に検証する。GET リクエストは body を持たないため `req.body` チェックで素通りし影響なし。バグレポート指定の全フィールド（`name`, `address`, `note`, `staff`, `customer_name`, `project_name`, `building_type`, `source`, `work_type`, `billing_type`）を網羅。`/\?{2,}/` は 2 文字以上の連続疑問符のみを検出するため、正常な日本語テキストや単一の `?`（疑問符を含む note 等）は誤検出されない。
