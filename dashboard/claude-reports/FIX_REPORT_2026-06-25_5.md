## 修正詳細

### Fix #1 — `完了` ステータスが案件編集フォームに存在しない
- **File:** `dashboard/public/index.html`
- **Lines affected:** `#prj-status` select（726行付近）
- **Root cause:** `#prj-status` の `<select>` には 6 つの option（初回訪問済 / 現地調査済 / 商談中 / 見積提出済 / 契約済 / 失注）しか定義されておらず、`完了` が欠落していた。DB・API は `完了` を受け付けるが、UI 上に選択肢が無いため、ユーザーが案件を完了に遷移させる導線が存在しなかった。
- **Fix applied:** `契約済` の直後に `<option value="完了">完了</option>` を追加し、業務ワークフロー順（…→契約済→完了→失注）を維持した。
  ```html
  <option value="契約済">契約済</option>
  <option value="完了">完了</option>   <!-- 追加 -->
  <option value="失注">失注</option>
  ```
- **Side-effect check:** option 追加のみ。`openProjectModal()` は `p.status` をそのまま `.value` に代入するため、既存 `完了` 案件の編集時も正しく選択状態となる。サーバーのパイプライン SQL は元々 `WHERE status NOT IN ('契約済','失注','完了')` で `完了` を除外済みのため、バックエンド変更は不要。回帰なし。

### Fix #2 — ISO形式 `contract_date` で月別チャートのラベルが壊れる
- **File:** `dashboard/server.js`
- **Lines affected:** `GET /api/projects/monthly`（80–95行付近）
- **Root cause:** 月別集計 SQL が `substr/instr` で `/` を区切りに年・月トークンを抽出していた。CRUD フォーム経由で保存された案件は HTML date picker の値（ISO `YYYY-MM-DD`）で格納されるため、`instr('2026-06-25','/')` が 0 を返し、`substr(...,1,-1)` が空文字となって `"-2026"` という不正ラベルが生成され、全 ISO 日付案件が 1 つの壊れたバケットにまとまっていた。
- **Fix applied:** `CASE WHEN contract_date LIKE '%/%'` で日付形式を分岐。スラッシュ形式は従来の printf 方式（メモリ上「動作確認済み」の手法）を維持し、ISO 形式は `substr(contract_date, 1, 7)` で直接 `YYYY-MM` を得る。
  - エージェントメモリ `project_date_storage_format.md` の警告に従い、`strftime('%Y-%m', REPLACE(date,'/','-'))` 方式（NULL を返す既知の失敗）は採用しなかった。
- **Side-effect check:** スラッシュ形式（既存 Excel インポートデータ）の出力は従来と完全一致。ISO 形式のみ新たに正しく `YYYY-MM` 化される。`node --check` 構文チェック合格。回帰なし。

### Fix #3 — モーダル表示中に背景スクロールがロックされない
- **File:** `dashboard/public/index.html`
- **Lines affected:** `openStaffModal()`, `openCustomerModal()`, `openProjectModal()`, `openEmployeeModal()`, `openInvoiceModal()` の各 `.classList.add('open')` 直後／`closeFormModal()`／オーバーレイクリック・staffModal の各クローズ経路
- **Root cause:** モーダルを開いても `document.body.style.overflow` が `visible` のままで、背景の `<main>` がスクロール可能だった。長いテーブルページでは送信ボタンへの `scrollIntoView` 時に背景が動き、ポインタイベントが背後の `<td>` に奪われる事象が発生していた。
- **Fix applied:**
  - 5 つのモーダル開関数（CRUD 4 種 + staffModal）に、`.classList.add('open')` 直後 `document.body.style.overflow = 'hidden';` を追加。
  - 全クローズ経路で `document.body.style.overflow = '';` を復元：`closeFormModal(id)`、`.modal-overlay` 背景クリックハンドラ、staffModal の保存完了時・×ボタン・背景クリックの 3 経路。
- **Side-effect check:** すべての開経路に対し復元経路が対応していることを確認（CRUD 4 種は `closeFormModal` + 背景クリックで復元、staffModal は専用 3 経路で復元）。閉じ忘れによる `overflow:hidden` 固着は無い。回帰なし。

### Fix #4 — 3つの「追加」ボタンに `id` 属性が無い
- **File:** `dashboard/public/index.html`
- **Lines affected:** 案件追加ボタン（464行付近）/ 請求追加ボタン（518行付近）/ 顧客追加ボタン（542行付近）
- **Root cause:** 3 つの追加ボタンが class と inline `onclick` のみで識別されており、安定した `id` セレクタによる自動テスト・支援技術ターゲティングが不可能だった。
- **Fix applied:** 各ボタンに一意の `id` を付与。`onclick` ハンドラはそのまま維持（最小変更）。
  - `id="btn-add-project"`（案件管理）
  - `id="btn-add-invoice"`（請求・入金管理）
  - `id="btn-add-customer"`（顧客台帳）
- **Side-effect check:** 属性追加のみ。既存の class・inline onclick・レイアウト（請求ボタンの `style="margin-left:auto"`）は不変。回帰なし。

### Fix #5 — `/api/projects/by-status` が契約金額でなく見積金額を返す
- **File:** `dashboard/server.js`
- **Lines affected:** `GET /api/projects/by-status`（67行付近）
- **Root cause:** SQL が全ステータス一律で `SUM(estimate_amount)` を集計していたため、契約済案件で見積金額と契約金額の差分（例：田中様邸 1,980,000 − 1,950,000 = 30,000円）が KPI カード（`contracted_total`）と by-status チャートの間で食い違っていた。
- **Fix applied:** `SUM(estimate_amount)` を `SUM(COALESCE(contract_amount, estimate_amount))` に変更。契約金額がある場合はそれを用い、無い（契約前ステージ）場合のみ見積金額にフォールバックする。これは同ファイルの `/api/projects/by-type`（75行）と同一パターン。
  - `COALESCE` は NULL 時のみ切替えるため、契約金額 `0` を欠損扱いしない（Falsy 値クォーク準拠）。
- **Side-effect check:** 契約済案件の集計が KPI カードと一致するようになる。契約前ステータスは従来どおり見積金額を表示。`COUNT(*)` ・グループ化・並び順は不変。`node --check` 構文チェック合格。回帰なし。
